# 01 总体架构

## 1. 技术选型

### 1.1 后端 —— 基于 pig

| 层面 | 选型 | 说明 |
| --- | --- | --- |
| 基础框架 | pig（Spring Boot 3.x + Spring Cloud + Spring Cloud Alibaba） | JDK 17+，直接复用 pig 的工程骨架与公共组件 |
| 认证授权 | Spring Authorization Server（pig-auth） | OAuth2.1，支持密码/授权码/客户端模式；工位终端使用专用客户端 |
| 网关 | Spring Cloud Gateway（pig-gateway） | 路由、限流、统一鉴权 |
| 注册/配置中心 | Nacos | 服务发现 + 动态配置 |
| ORM | MyBatis-Plus | 沿用 pig 代码生成体系 |
| 数据库 | MySQL 8.x（主库） | 业务数据；分库策略见 §4 |
| 时序数据 | TDengine 或 TimescaleDB（P1 引入） | 设备过程数据、能源数据等高频时序量 |
| 缓存 | Redis | 会话、看板实时数据、分布式锁 |
| 消息 | RocketMQ（或 Kafka） | 现场事件流、模块间解耦、ERP 异步集成 |
| 任务调度 | XXL-Job（pig 生态内） | 班次切割、KPI 汇总、接口轮询 |
| 设备接入 | 独立采集服务（OPC UA: Eclipse Milo；MQTT: EMQX） | 见 [10 集成需求](10-集成需求.md) |
| 实时推送 | WebSocket / SSE | 看板、安灯、排程甘特图实时刷新 |
| 文件存储 | pig 内置 OSS 模块（S3 协议，MinIO） | 作业指导书、检验图片、文档 |
| 报表 | 内置图表（ECharts）+ 预留 JimuReport 集成 | 驾驶舱与分析 |

### 1.2 前端 —— 基于 vue-vben-admin v5

| 层面 | 选型 | 说明 |
| --- | --- | --- |
| 框架 | Vue 3 + TypeScript + Vite + Turbo monorepo | 采用 vben v5 的 `apps/ + packages/` 结构 |
| UI 组件 | Ant Design Vue 版本（`apps/web-antd`） | 与 pig 生态前端习惯一致 |
| 状态/路由 | Pinia + Vue Router（vben 内置） | 动态菜单与按钮级权限对接 pig RBAC |
| 甘特图 | 自研基于 Canvas/SVG 的排程甘特组件（或集成 dhtmlx-gantt 商业版，需决策） | 电子排程看板核心交互 |
| 大屏/看板 | 独立轻量页面（全屏路由），ECharts + WebSocket | 车间电视/LED 大屏 |
| 工位终端 | 独立 app（`apps/web-terminal`）：大按钮触屏 UI、扫码优先、可离线缓存 | 与管理端共享 packages 但独立交付 |

> **前端形态结论**：一个 monorepo、三个应用 —— `web-antd`（管理端）、`web-terminal`（工位终端）、`web-dashboard`（看板大屏）。工位终端不复用管理端布局，按"全屏、大字、少层级、扫码驱动"专门设计。

## 2. 服务划分

在 pig 既有服务（`pig-auth`、`pig-gateway`、`pig-upms`、`pig-monitor` 等）基础上，新增业务服务。**首期建议采用"少服务、多模块"的布局**，避免过早微服务化带来的分布式事务负担；服务内部按业务域分 Maven 模块，保留拆分缝：

```
xmes
├── pig-auth / pig-gateway / pig-upms ...          # pig 原生服务（用户/角色/菜单/租户）
├── xmes-modeling      车间建模与制造主数据服务      # 工厂结构、物料、BOM、工艺路线、班次日历
├── xmes-execution     生产执行服务                 # 工单、作业、排程、报工、安灯、交接班
├── xmes-resource      资源服务                    # 设备、维护、工装模具、检具、NC程序、能源
├── xmes-material      物料与追溯服务               # WIP、线边库、批次/序列号、配送、追溯链
├── xmes-quality       质量服务                    # 检验计划/单、SPC、不合格品、投诉
├── xmes-hr            人员服务                    # 工时、排班、资质（依赖 pig-upms 用户）
├── xmes-analytics     分析服务                    # KPI 汇总、OEE、驾驶舱查询、升级与措施
├── xmes-connect       设备接入服务                 # OPC UA/MQTT/Modbus 采集、协议适配、边缘缓存
└── xmes-integration   集成服务                    # ERP 适配器、开放 API、Webhook
```

划分依据：

- `xmes-connect` 必须独立：高频 IO、可多实例就近部署在车间，故障不影响管理功能；
- `xmes-execution` 是写入热点（报工、状态事件），独立伸缩；
- `xmes-analytics` 只读汇总，可指向只读副本；
- 其余服务首期可合并部署（同一进程多模块），按负载再拆。

### 3.1 事件驱动约定

现场行为统一建模为**生产事件**（领域事件），写入事件总线后由各订阅方消费：

| 事件类别 | 示例 | 生产者 | 主要消费者 |
| --- | --- | --- | --- |
| 作业事件 | 开工、暂停、完工、报数（良/废/返） | execution | analytics、material、integration(ERP回传) |
| 设备事件 | 状态变更、报警、计数脉冲 | connect | execution(自动报工)、resource、analytics |
| 质量事件 | 检验完成、判不合格、SPC 违规 | quality | execution(拦截)、analytics、升级引擎 |
| 物料事件 | 上料、移转、批次绑定 | material | quality(追溯)、analytics |
| 人员事件 | 上岗/离岗、刷卡 | hr | execution(人员-作业关联)、analytics |

事件不可变、带全局序号与发生时间，KPI 与追溯链均可由事件流重放重建。

## 4. 数据架构要点

- **多租户**：沿用 pig 的租户体系（tenant_id 行级隔离），支持集团多工厂——租户=法人/工厂集团，工厂/车间为租户内建模对象。
- **业务库分库**：每个业务服务独立 schema；跨服务仅通过 API/事件交互，禁止跨库 join。
- **冷热分离**：报工/事件明细按月分表（或分区），保留期可配置（默认在线 24 个月）；时序过程数据进时序库。
- **主数据归属**：物料、BOM、工艺以 ERP/PLM 为源（可配置为 MES 自维护），MES 侧只读 + 车间扩展属性；工厂结构、设备、班次为 MES 自有主数据。
- **编码规则引擎**：批次号、序列号、工单号等支持可配置编码规则（常量/日期/序列/校验位段）。

## 5. 权限模型（基于 pig-upms 扩展）

| 维度 | 实现 |
| --- | --- |
| 功能权限 | pig 菜单/按钮权限（RBAC） |
| 数据权限 | 扩展"组织范围"维度：用户绑定 工厂/车间/产线 节点，业务查询按节点过滤 |
| 工位权限 | 工位终端按"工位登录 + 人员上岗"双因素：终端绑定工位，人员刷卡/扫码上岗后才可报工 |
| 资质拦截 | 作业要求资质时，报工人员须持有有效资质（见 07 人员管理） |
| 审批 | 关键流程（不合格品处置、工艺参数放行、补报工）走简化审批流（首期两级审批，预留对接工作流引擎） |

## 6. 部署形态

- **标准部署**：私有化，单 K8s 集群或 Docker Compose（中小车间），所有服务集中部署。
- **车间边缘**：`xmes-connect` 可下沉到车间边缘服务器，断网时本地缓存采集数据，恢复后补传（store-and-forward）。
- **高可用**：网关/业务服务多副本，MySQL 主从，Redis 哨兵；详见 [11 非功能需求](11-非功能需求.md)。
