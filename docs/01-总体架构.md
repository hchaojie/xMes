## 2. 平台理念：对标 MPDV MIP

依据《MIP 交流》文档，MPDV 的平台架构核心是：**ViPR（虚拟生产现实）作为生产的数字镜像，所有 mApp 通过 CAS 读写统一的 MBO（制造业务对象），现场事件通过 SCS（含校验引擎+处理引擎）进入平台，企业系统通过 ECS 适配器集成，设备通过 DEC 边缘组件接入**。xmes 采用同构的分层思想：

| MIP 概念 | 职责 | xmes 对应实现 |
| --- | --- | --- |
| ViPR + MBO | 统一对象模型，生产的数字镜像（事件 Events / 在线投影 Online Projection / 结果记录 Results） | 02 车间对象模型 + 事件流架构（§4.1）：不可变事件 → 实时状态投影 → 班次结果记录三层数据 |
| CAS 内容访问服务 | mApp 读写 MBO 的标准服务 | 各服务暴露的统一 REST API（OpenAPI 3），业务模块只经 API/事件交互 |
| SCS 车间连接服务 | 采集现场事件（人工/半自动/自动），含**校验引擎**（权限/引用/业务合理性校验）与**处理引擎**（事件→投影→结果记录） | xmes-execution 的报工接入层：统一事件入口 + 可配置防错校验链（FR-SHF-31）+ 事件消费更新投影 |
| ECS 企业连接服务 | 业务系统适配器（REST/SOAP/SAP RFC/文件/直连 DB），传输表机制、分发模型 | xmes-integration（FR-INT-1x、3x） |
| DEC 分布式边缘计算 | 设备协议接入、信号预处理（逻辑模块/虚拟点）、计数/节拍监控、DNC 上传下载、MQTT 数据分发 | **后期独立物联网模块**（FR-INT-2x，MES 侧预留事件契约，见 §3） |

## 3. 服务划分

在 pig 既有服务（`pig-auth`、`pig-gateway`、`pig-upms`、`pig-monitor` 等）基础上新增业务服务。**已确认首期聚焦 3 个业务服务**（决策 Q5/Q6：设备数据采集移出 MES，由后期独立物联网模块承担），服务内部按业务域分 Maven 模块，保留拆分缝：

```
xmes
├── pig-auth / pig-gateway / pig-upms ...   # pig 原生服务（用户/角色/菜单/租户）
│
├── xmes-core          平台与建模服务（≈ ViPR/CAS）
│   ├── modeling   工厂结构、班次日历、编码规则
│   ├── masterdata 物料、BOM、工作计划(工艺路线)、工艺参数模板   ← BOM/工艺以 MES 为主
│   ├── resource   设备/工装/检具/NC程序 台账与状态模型
│   ├── document   受控文档
│   └── integration ERP 适配器（≈ ECS）
│
├── xmes-execution     生产执行服务（≈ SCS + 订单/人员 mApps）
│   ├── order      工单、作业、排程(自研甘特后端)、派工
│   ├── booking    报工事件入口(校验引擎)、工时票、结果记录
│   ├── shopfloor  工位终端后端、安灯、交接班
│   ├── hr         上岗工时、排班、资质
│   └── analytics  KPI 结算、驾驶舱查询（二期视负载拆出）
│
├── xmes-quality-material  质量与物料服务
│   ├── quality    检验计划/单、SPC、不合格品、投诉
│   └── material   批次/序列号、缓存区库存、配送、追溯谱系
│
└── [后期·独立项目] 物联网模块（≈ DEC）：设备协议采集、边缘缓存、时序存储
        ↳ MES 侧本期只预留：DataTag 对象、设备事件接入 API（事件信封契约）、自动报工挂点
```

划分依据：

- `xmes-execution` 是写入热点（报工、人工状态事件），独立伸缩；
- `xmes-core` 聚合低频写、高频读的主数据与台账，是其他服务的对象字典；ERP 集成与主数据同源，并入 core；
- `xmes-quality-material` 二者共享批次对象与追溯链，耦合最紧，合并降低跨服务事务；
- analytics、integration 首期以模块形态并入宿主服务，接口与事件契约按独立服务设计，二期可平移拆出；
- 物联网模块是独立的后期项目：MES 通过"设备事件接入 API"消费其数据（状态事件、计数、过程值），双方以事件契约解耦——物联网模块上线前，设备状态由终端人工切换维护。

### 4.1 事件驱动约定

现场行为统一建模为**生产事件**（领域事件）。首期分发实现：事件随业务事务写入 outbox 事件表，由分发器投递到 Redis Stream 供订阅方消费（at-least-once + 消费幂等）；事件信封格式按消息中间件标准设计，后续切换 RocketMQ 时订阅方无感：

| 事件类别 | 示例 | 生产者 | 主要消费者 |
| --- | --- | --- | --- |
| 作业事件 | 开工、暂停、完工、报数（良/废/返） | execution | analytics、material、integration(ERP回传) |
| 设备事件 | 状态变更、报警、计数脉冲 | 首期：终端人工切换；后期：物联网模块经设备事件接入 API 写入 | execution(自动报工)、analytics |
| 质量事件 | 检验完成、判不合格、SPC 违规 | quality | execution(拦截)、analytics、升级引擎 |
| 物料事件 | 上料、移转、批次绑定 | material | quality(追溯)、analytics |
| 人员事件 | 上岗/离岗、刷卡 | hr | execution(人员-作业关联)、analytics |

事件不可变、带全局序号与发生时间，KPI 与追溯链均可由事件流重放重建。

## 5. 数据架构要点

- **多租户**：沿用 pig 的租户体系（tenant_id 行级隔离），支持集团多工厂——租户=法人/工厂集团，工厂/车间为租户内建模对象。
- **业务库分库**：每个业务服务独立 schema；跨服务仅通过 API/事件交互，禁止跨库 join。
- **冷热分离**：报工/事件明细按月分表（或分区），保留期可配置（默认在线 24 个月）；高频过程数据/时序存储归属后期物联网模块，MES 不存原始时序量。
- **主数据归属**：物料、BOM、工艺以 ERP/PLM 为源（可配置为 MES 自维护），MES 侧只读 + 车间扩展属性；工厂结构、设备、班次为 MES 自有主数据。
- **编码规则引擎**：批次号、序列号、工单号等支持可配置编码规则（常量/日期/序列/校验位段）。

## 6. 权限模型（基于 pig-upms 扩展）

| 维度 | 实现 |
| --- | --- |
| 功能权限 | pig 菜单/按钮权限（RBAC） |
| 数据权限 | 扩展"组织范围"维度：用户绑定 工厂/车间/产线 节点，业务查询按节点过滤 |
| 工位权限 | 工位终端按"工位登录 + 人员上岗"双因素：终端绑定工位，人员刷卡/扫码上岗后才可报工 |
| 资质拦截 | 作业要求资质时，报工人员须持有有效资质（见 07 人员管理） |
| 审批 | 关键流程（不合格品处置、工艺参数放行、补报工）走简化审批流（首期两级审批，预留对接工作流引擎） |

## 7. 部署形态

- **标准部署**：私有化，单 K8s 集群或 Docker Compose（中小车间），所有服务集中部署。
- **车间边缘**：边缘部署与采集断网缓存（store-and-forward）属于后期物联网模块的部署形态，MES 本期不涉及。
- **高可用**：网关/业务服务多副本，MySQL 主从，Redis 哨兵；详见 [11 非功能需求](11-非功能需求.md)。
